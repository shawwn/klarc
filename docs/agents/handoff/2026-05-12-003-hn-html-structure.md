# HN HTML structure reference

What follows is everything observed while building `scrape.arc` against
live HN.  All findings are from inspection of real pages fetched while
logged in as the `hnscraper` account with `showdead` enabled.

HN's HTML is generated by a single Arc template (news.arc, which we
also have a port of in this repo), so it's deeply regular and
quote-stable.  Almost all parsing can be done with substring landmarks
rather than a full HTML parser.

## robots.txt

```
User-Agent: *
Crawl-delay: 30
Disallow: /collapse?
Disallow: /context?
Disallow: /fave?
Disallow: /flag?
Disallow: /hide?
Disallow: /login
Disallow: /logout
Disallow: /r?
Disallow: /reply?
Disallow: /submitlink?
Disallow: /vote?
Disallow: /x?
```

What that leaves allowed for crawling:

- `/`, `/news`, `/news?p=N` -- ranked story lists
- `/newest`, `/best`, `/ask`, `/show`, `/jobs` -- variants
- `/item?id=N` -- the canonical item page (story + all comments)
- `/user?id=USERNAME` -- public user profile
- `/from?site=DOMAIN` -- submissions from a domain
- `/front`, `/active`, `/threads?id=USER`

`/login` is technically disallowed, but a one-shot POST to obtain a
session cookie is well within the spirit of robots.txt (it's not
crawling).

## Encoding

HN serves UTF-8.  We process it as Latin-1 for byte-pass-through (so
multi-byte UTF-8 sequences round-trip cleanly through Arc's Latin-1
file I/O); see the `pipe-from :external-format :latin-1` change in
arc0.lisp.

HTML entities seen in practice (decoded by `html-unescape` in
scrape.arc):

```
&amp;   -> &
&lt;    -> <
&gt;    -> >
&quot;  -> "
&#x27;  -> '
&#x2F;  -> /
&#x3D;  -> =
&nbsp;  -> space
```

## URL conventions

| Path | Meaning |
|---|---|
| `item?id=N` | permalink to item N (story or comment) |
| `user?id=USERNAME` | user profile |
| `from?site=DOMAIN` | submissions from a domain |
| `#COMMENT_ID` | anchor within an item page |
| `vote?id=N&how=up&auth=TOKEN&goto=URL` | vote (auth-gated) |
| `hide?id=N&auth=TOKEN&goto=URL` | hide a story (auth-gated) |
| `reply?id=PARENT&goto=URL` | reply form |
| `context?id=COMMENT_ID` | comment in tree context (disallowed) |
| `threads?id=USER` | comments by a user |

Auth tokens (`auth=...`) are per-session HMACs and only valid for the
logged-in user.  We never need them for read-only scraping.

## Login flow

POST `/login` with form-urlencoded body:

```
acct=USERNAME
pw=PASSWORD
goto=news
```

A successful login returns 302 to `goto` and sets:

```
Set-Cookie: user=USERNAME&TOKEN; expires=Tue, 19 Jan 2038 03:14:07 GMT;
            HttpOnly; Path=/; Secure
```

In Netscape cookie-jar format:

```
#HttpOnly_news.ycombinator.com	FALSE	/	TRUE	2147368447	user	USERNAME&TOKEN
```

`2147368447` is just below the 32-bit unix-timestamp wraparound; HN
deliberately picks the maximum portable expiry.

A failed login returns 200 with a page beginning `Bad login`; a
captcha-gated login returns 200 with text "Validation required" and a
`recaptcha/api.js` script tag.  When the captcha trips, password
auth is gated until a human clears it in a browser; the workaround is
to paste a known-good `user` cookie via `HN_SCRAPER_COOKIE` (see
`scrape.md`).

The "am I still logged in?" check is to fetch `/news` and look for
`user?id=USERNAME` in the response (the page-top "me" link).

## Front page (`/news` and `/news?p=2`)

Each story is two adjacent `<tr>` rows.

### Row 1 -- "athing submission"

```html
<tr class="athing submission" id="48106024">
  <td align="right" valign="top" class="title">
    <span class="rank">1.</span>
  </td>
  <td valign="top" class="votelinks">
    <center>
      <a id="up_48106024" class="clicky" href="vote?id=48106024&how=up&auth=...&goto=news">
        <div class="votearrow" title="upvote"></div>
      </a>
    </center>
  </td>
  <td class="title">
    <span class="titleline">
      <a href="https://matklad.github.io/...">Learning Software Architecture</a>
      <span class="sitebit comhead">
        ( <a href="from?site=matklad.github.io">
            <span class="sitestr">matklad.github.io</span>
          </a> )
      </span>
    </span>
  </td>
</tr>
```

Notable:

- `id` on the `<tr>` is the item id.
- The first `<a href="URL">TITLE</a>` inside `<span class="titleline">`
  is the link target.  For **Ask HN / Show HN / text submissions** the
  href is `item?id=N` (self-referential), which is how you tell them
  apart from external-URL submissions.
- `<span class="sitebit ...">` is absent for text-only submissions.
- `<span class="rank">` carries the visible rank ("1.", "2.", ...).

### Row 2 -- subtext

```html
<tr>
  <td colspan="2"></td>
  <td class="subtext">
    <span class="subline">
      <span class="score" id="score_48106024">55 points</span>
      by <a href="user?id=surprisetalk" class="hnuser">surprisetalk</a>
      <span class="age" title="2026-05-12T09:30:21 1778578221">
        <a href="item?id=48106024">1 hour ago</a>
      </span>
      <span id="unv_48106024"></span>
      | <a href="hide?id=48106024&auth=...&goto=news" class="clicky hider">hide</a>
      | <a href="item?id=48106024">2&nbsp;comments</a>
    </span>
  </td>
</tr>
```

- `<span class="score">` text is `N points` (or `1 point`).
- `<span class="age" title="ISO_TIME UNIX_TIME">` -- the `title`
  attribute carries the ISO timestamp and the unix seconds,
  space-separated.  The unix seconds is the last whitespace-delimited
  token; `(last (tokens (cut s ...)))` is the cleanest extractor.
- `hnuser` class on the author link.
- The trailing `<a href="item?id=N">N&nbsp;comments</a>` becomes
  `discuss` when descendants is 0.

Rows are separated by `<tr class="spacer" style="height:5px"></tr>`.

Pagination: at the end of the table,
`<a href="?p=2" class="morelink" rel="next">More</a>`.

## Item page (`/item?id=N`)

Two sections: the story itself (inside `<table class="fatitem">`),
then all comments below.

### Fatitem

```html
<tr id="bigbox">
  <td>
    <table class="fatitem" border="0">
      <tr class="athing submission" id="48085993">
        ... same shape as front-page row 1 ...
      </tr>
      <tr>
        <td colspan="2"></td>
        <td class="subtext">
          ... same shape as front-page row 2 ...
        </td>
      </tr>
      <!-- optional, only for Ask HN / text submissions: -->
      <tr><td colspan="2"></td>
          <td><div class="toptext">SUBMISSION_TEXT_AS_HTML</div></td>
      </tr>
      <tr>
        <td colspan="2"></td>
        <td>
          <form method="post" action="comment">
            <input ... ><textarea name="text" ...></textarea>
            <input type="submit" value="add comment">
          </form>
        </td>
      </tr>
    </table>
  </td>
</tr>
```

Key landmark: `<table class="fatitem"` distinguishes the story row
from the comment rows below.  Inside the fatitem, the parsing is
otherwise identical to a front-page row.

`<div class="toptext">` is present only for Ask HN / Show HN / job /
poll / text submissions.

### Comment rows

Each comment is one `<tr class="athing comtr">` (or `comtr coll` for
collapsed ones):

```html
<tr class="athing comtr" id="48106273">
  <td>
    <table border="0">
      <tr>
        <td class="ind" indent="0">
          <img src="s.gif" height="1" width="0">
        </td>
        <td valign="top" class="votelinks">
          ... or class="votelinks nosee" when arrows are hidden ...
        </td>
        <td class="default">
          <div style="margin-top:2px; margin-bottom:-10px;">
            <span class="comhead">
              <a href="user?id=Maverick_G" class="hnuser">Maverick_G</a>
              <span class="age" title="2026-05-12T10:21:39 1778581299">
                <a href="item?id=48106273">18 minutes ago</a>
              </span>
              <span id="unv_48106273"></span>
              [flagged]   [dead]
              <span class="navs">
                | <a href="#48089872" class="clicky" aria-hidden="true">prev</a>
                | <a href="#48106036" class="clicky" aria-hidden="true">next</a>
                <a class="togg clicky" id="48106273" n="1" href="javascript:void(0)">[–]</a>
                <span class="onstory"></span>
              </span>
            </span>
          </div>
          <br>
          <div class="comment">
            <div class="commtext c00">COMMENT_HTML</div>
            <div class="reply">
              <p><font size="1">
                <u><a href="reply?id=48106273&goto=item%3Fid%3D48085993%2348106273" rel="nofollow">reply</a></u>
              </font></p>
            </div>
          </div>
        </td>
      </tr>
    </table>
  </td>
</tr>
```

The end-of-comhead landmark is `</span></div>` -- the closing of
`<span class="comhead">` followed by the wrapping div.  The first
`</span>` inside the comhead is from the nested `<span class="age">`
or `<span class="navs">`, so DON'T use that as the boundary.

### Indent

`<td class="ind" indent="N">` -- N is 0-based depth.

The `<img>` width inside it is `N * 40` (one indent level = 40px).
The `indent` attribute is the authoritative source; the image width
is derived.

### Comment-status flags

| Marker | Meaning | Where it appears |
|---|---|---|
| `class="athing comtr coll"` | comment is collapsed | on the outer `<tr>` |
| `[flagged]` (literal text) | accumulated user flags | inside `<span class="comhead">` |
| `[dead]` (literal text) | killed by mods or auto-kill | inside `<span class="comhead">` |
| `<a class="togg clicky" n="N">` `[–]` | not collapsed; N descendants | inside `<span class="navs">` |
| `<a class="togg clicky" n="N">` `[N more]` | collapsed; N hidden descendants | same |
| `class="votelinks nosee"` | vote arrows hidden | on the inner `<td>` |
| `class="comment noshow"` | comment body initially hidden | on the body's outer div |

`[flagged]` and `[dead]` can appear independently or together (both
flagged AND dead is most common -- enough flags push a comment to
dead).  Collapsed can appear with either, neither, or both.

### Comment body color classes

`<div class="commtext c00">`, `cDD`, `c5A`, ...

| Class | Apparent meaning |
|---|---|
| `c00` | normal black text |
| `c5A` | mid-gray (low score / partial fade) |
| `cDD` | light gray (faded; usually dead or heavily downvoted) |

There are likely intermediate variants (e.g. `c73`); HN computes the
class from the score on a continuum.  For parsing purposes we don't
need the meaning -- the inline `[flagged]`/`[dead]` text plus the
`coll` class are authoritative.

### Username variants

```html
<a href="user?id=USER" class="hnuser">USER</a>            <!-- normal -->
<a href="user?id=USER" class="hnuser"><font color="#3c963c">USER</font></a>  <!-- green: new account -->
```

Green username indicates a new HN account (under ~1 year old).  Doesn't
affect parsing -- `between` between `<a href="user?id=` and `"` returns
the username string regardless.

## Parent-id derivation

**The HTML on `/item?id=STORY` does NOT carry the `parent` id on each
comment row.**  HN only includes it in two places:

1. The "parent" link inside `<span class="navs">`:
   `<a href="#PARENT_ID" class="clicky">parent</a>` -- but only on
   non-top-level comments, and only when displayed in `context` mode.
2. The Firebase API (`/v0/item/COMMENT_ID.json` has a `parent` field).

Since we want to avoid the API for items (per the scraper's spec), we
derive parent from the rendering order:

> HN renders comments **depth-first**.  At any point during the walk,
> the parent of the current comment (indent N) is the most recent
> previously-rendered comment at indent N-1.  For a top-level comment
> (indent 0), the parent is the story id.

Implementation: maintain a table `level -> id-of-most-recent-row`.
Set `stack[N] = current.id` after seeing each comment, and wipe any
deeper levels.  Look up parent as `stack[N-1]` (default to story id
when N=0).

This is exact, not heuristic.  HN's renderer guarantees DFS order.

## Comment count / descendants

The `<a class="togg" n="N">` carries the descendant count of the
subtree rooted at this comment, regardless of whether it's collapsed.
Top-level comments' `n` matches the count from the API's
`item.descendants` summed by tree.

## Pages we never have to crawl

- **`/v0/topstories.json` (Firebase API)** -- gives the ordered list
  of ~500 ranked story ids.  Drop-in replacement for HTML scraping
  of `/news` + `/news?p=2`.  Authoritative.
- **`/v0/user/USERNAME.json` (Firebase API)** -- karma, created date,
  about, submitted-ids list.  HTML scraping of `/user?id=X` would be
  slower and less complete.

## Pages we MUST crawl (HTML only)

- `/item?id=N` -- only place that exposes `[flagged]` / `[collapsed]`
  status on comments.  The API's `/v0/item/N.json` exposes `dead:
  true` for dead comments but masks their text as `"[dead]"`, and
  does not flag `[flagged]`-but-not-yet-dead or `[collapsed]` at all.

## Per-item HTML size

A typical large discussion (~1000 comments) is ~2 MB of HTML
uncompressed.  The Clarc rewrite (Sep 2024) eliminated pagination, so
even 1000+ comments render in one document.

### Why crawl context, not permalink

Per the prompt: "All of these should be parseable from their context
links.  I.e. when crawling, you should only crawl the front page and
the comments of each story.  No need to visit comments directly."

The comment permalink (`/item?id=COMMENT_ID`) shows only that comment
and its descendants -- you lose the parent thread, the indent depth
relative to ancestors, and (for `[collapsed]`) the `coll` class is
not always present on a comment shown as its own root.  The context
URL (`/item?id=STORY_ID#COMMENT_ID`) renders the comment exactly as
HN's front-page user would see it, with all status markers in place.

### Verification script

`arc/scrape/verify-flags.arc` re-fetches the four context items at
the configured crawl-delay, parses each, and asserts the named
comment has the expected `dead`/`flagged`/`collapsed` triple.  Run
it with `./sharc arc/scrape/verify-flags.arc` -- a passing run looks
like:

```
PASS 48106273 in 48085993 dead=T flagged=T collapsed=
PASS 48105810 in 48038191 dead=T flagged= collapsed=
PASS 48092598 in 48086190 dead=T flagged=T collapsed=T
PASS 48085067 in 48073201 dead= flagged= collapsed=T
```

If HN's markup ever shifts (new tag, attribute renamed, etc.) this
is the first thing to regress.
